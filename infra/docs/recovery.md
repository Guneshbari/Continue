# Continue — Disaster Recovery Playbook

This document details the step-by-step pipeline to safely restore a database snapshot.

---

## 1. Restoration Procedure

To prevent partial schema tables or inconsistent relational states, the restore pipeline uses the `--single-transaction` parameter. 

> [!CAUTION]
> **Transaction Boundaries Limitation**
> Restoring with `--single-transaction` wraps all commands in a single transaction block. If your dump contains schema commands that cannot be run inside a transaction (such as `VACUUM` or certain concurrent index builds), postgres will throw an error and rollback. Dumps generated via Continue's standard `backup.sh` are fully transactional and safe to restore this way.

### Step 1: Locate and Uncompress Snapshot
Unzip the selected `.sql.gz` snapshot from the local backup directory:
```bash
gunzip -c ./backups/continue-backup-YYYY-MM-DD_HH-MM-SS.sql.gz > temp_restore.sql
```

### Step 2: Enforce Single-Transaction Restore
Restore the database using `psql` or `pg_restore` (depending on dump format). For SQL text dumps generated by standard pg_dump:
```bash
psql "${DATABASE_URL}" --single-transaction -f temp_restore.sql
```

### Step 3: Cleanup Temporary Files
Delete the unzipped SQL file immediately to protect credentials or user metrics:
```bash
rm temp_restore.sql
```

---

## 2. Recovery Verification Checklist

Never stop at "restore completed" messages. Complete these post-restoration checks to ensure total data integrity:

- [ ] **Verify Tables Exist**: Inspect the schema using `\dt` in psql to confirm `users`, `games`, `ratings`, and `reviews` tables are fully restored.
- [ ] **Verify Row Counts**: Compare user and game row counts against pre-incident logs to confirm no rows were omitted.
- [ ] **Validate Database Migrations**: Run the database status check command to confirm all Prisma migrations are marked active:
  ```bash
  npx prisma migrate status
  ```
- [ ] **Confirm Uptime & Liveness**: Query the liveness endpoint to confirm process integrity:
  ```bash
  curl http://localhost:3001/api/v1/health
  ```
- [ ] **Confirm Database Readiness**: Query the database connectivity ping endpoint to verify NestJS can read data:
  ```bash
  curl http://localhost:3001/api/v1/health/ready
  ```
- [ ] **Smoke Test Core Flows**: Log in as a test user and verify review creation works to confirm active write permissions.
